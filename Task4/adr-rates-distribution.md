### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| NFR1 | Решение строится на текущем процессе работы со ставками (модуль ставок в АБС + публикация в Kafka из Задания 3) — без дублирования источника. |
| NFR2 | Внутренний кол-центр получает актуальные ставки близко к реальному времени (через API Сервиса ставок). |
| NFR3 | Партнёрский кол-центр — внешняя ИС; интеграция только файлами по SFTP (API-вызовы невозможны), односторонняя (банк → партнёр). |
| NFR4 | Определён SLA свежести ставок у партнёра (регулярная выгрузка по расписанию + выгрузка при изменении); в файле — отметка времени актуальности. |
| NFR5 | Безопасность передачи: SFTP с шифрованием канала и аутентификацией по ключам; контроль целостности файла (контрольная сумма/манифест). |
| NFR6 | Согласованный и версионируемый формат файла; устойчивость к ошибкам приёма (повторная доставка, идемпотентность по версии файла). |
| NFR7 | Подключение партнёрского кол-центра не должно нагружать АБС: ставки отдаются из Сервиса ставок, а не из БД АБС. |
| NFR8 | Наблюдаемость доставки: логирование выгрузок, подтверждение/контроль успешной передачи файлов партнёру. |
| NFR9 | Масштабируемость кол-центрового канала под рост числа обращений после маркетинговой кампании. |

### **Решение**

Диаграммы (в этой же папке):

- **Контекст** — `c4-context-rates-distribution.puml`;
- **Компоненты** (детализирован Сервис ставок) — `c4-component-rates-service.puml`.

#### Суть

Вводится новый **Сервис ставок (Rates Distribution Service)**, который переиспользует поток изменений ставок из Задания 3: модуль ставок АБС публикует события в **Kafka**, Сервис ставок их потребляет и поддерживает актуальное хранилище ставок. Он становится единой точкой раздачи актуальных ставок; витрина/read-model интернет-банка из Задания 3 потребляет тот же поток событий из Kafka, поэтому второго источника правды не возникает. Дальше — два канала раздачи:

- **Внутренний кол-центр (CRM)** — синхронно по **REST API** Сервиса ставок (UC2). Near-real-time, без обращения к АБС.
- **Партнёрский кол-центр** — асинхронно **файлами по SFTP** (UC3–UC4): отдельный компонент формирует файл актуальных ставок по расписанию и при значимом изменении, SFTP-коннектор доставляет его во внешнюю систему партнёра. Это единственно возможный для партнёра способ (нет API).

#### Логика решения

- **Единый источник** — ставки берутся из уже спроектированного потока (АБС → Kafka), без новой логики расчёта (NFR1).
- **Развязка от АБС** — оба кол-центра читают из Сервиса ставок, а не из перегруженной БД АБС (NFR7), что важно при росте звонков (NFR9).
- **Разные механизмы под разные ограничения** — внутренняя система умеет API (быстро, свежо), внешний партнёр умеет только файлы/SFTP (NFR2, NFR3).
- **Безопасность файлового канала** — SFTP, ключи, контроль целостности, отметка времени в файле как дисклеймер актуальности (NFR4, NFR5).

### **Альтернативы**

- **Прямой доступ CRM/партнёра к БД АБС или к API АБС.** Отклонено: нагрузка на перегруженную БД АБС, сильная связность, риск для работоспособности банка.
- **API для партнёрского кол-центра.** Невозможно: партнёр работает во внешней ИС и принимает только файлы (SFTP).
- **Партнёр читает напрямую из Kafka.** Отклонено: внешняя сторона, нет и не нужна интеграция брокера за периметр банка.
- **Передача файлов партнёру по электронной почте.** Отклонено: ниже безопасность и надёжность, нет гарантий доставки и целостности.
- **Ручная выгрузка XLS-файла оператором (как в текущем процессе «как есть»).** Отклонено: не масштабируется под рост обращений, ошибки ручного труда, неактуальность.
- **Отдельное хранилище ставок только для кол-центров (без переиспользования потока из Задания 3).** Отклонено: дублирование источника и риск рассинхронизации.

**Недостатки, ограничения, риски**

- **Задержка актуальности у партнёра.** Файловая интеграция даёт eventual-актуальность; риск озвучивания устаревших ставок — митигируется частотой выгрузки и отметкой времени в файле.
- **Партнёр только консультирует.** Оформление всё равно в банке/отделении; нужен чёткий скрипт перевода клиента.
- **Организационная зависимость от внешней стороны.** Требуются согласование формата, ключей и эксплуатации SFTP с партнёром.
- **Безопасность файла со ставками.** Риск утечки коммерческой информации; обязательны ключи, шифрование, ограничение доступа и аудит.
- **Новый сервис = операционная сложность.** Дополнительная команда, мониторинг доставки, версионирование формата.
- **Зависимость от готовности потока ставок (АБС → Kafka).** Сервис ставок не запустится раньше, чем АБС начнёт публиковать ставки (см. RoadMap).
