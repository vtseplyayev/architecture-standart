# FURPS+ — Цифровизация депозитов (MVP: заявка на депозит в интернет-банке и на сайте)

Архитектурно-значимые требования (АЗТ) выделены в комментарии. Требования, добавленные сверх текста задания, помечены **(доп.)** с обоснованием.

| Код | Требования                                                                                                                            | Комментарий                                                                                                                                                              |
|-----|---------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **F** | **Функциональные (Functionality)**                                                                                                  |                                                                                                                                                                         |
| F1  | Сайт: новый клиент видит список доступных депозитов с актуальными ставками.                                                            | Публичное чтение без авторизации. Источник ставок — не XLS (см. F8, S1). Витрина ставок должна быть доступна сайту без нагрузки на АБС.                                  |
| F2  | Сайт: подача заявки (ФИО + номер телефона), заявка попадает в систему кол-центра.                                                      | **АЗТ:** интеграция Сайт → CRM кол-центра. Точка входа нового клиента.                                                                                                  |
| F3  | Кол-центр: менеджер изучает заявку в системе кол-центра и может предложить особые условия по телефону.                                 | Переиспользуется существующий процесс/система кол-центра.                                                                                                                |
| F4  | Идентификация нового клиента в отделении (личная проверка документов) перед открытием депозита.                                        | Регуляторное бизнес-правило (KYC). На MVP онлайн-онбординг невозможен — определяет сценарий «сайт → звонок → отделение».                                                  |
| F5  | Интернет-банк: клиент видит список депозитов с актуальными ставками и персонализированные ставки лично для него.                       | **АЗТ:** нужен источник персонализированных ставок и идентифицированная сессия клиента (см. F11).                                                                        |
| F6  | Интернет-банк: клиент подаёт заявку, указав счёт и сумму; операция подтверждается СМС-кодом.                                           | **АЗТ:** интеграция с СМС-функционалом (см. F7) и со счётами клиента.                                                                                                    |
| F7  | СМС: бизнес-функционал работы с СМС реализуется силами команды банка, без доработки ядра подрядчиком.                                  | **АЗТ + ограничение (+R4):** новый сервис СМС-кодов/уведомлений строится вне ядра, ядро только как транспорт. Снижает зависимость от подрядчика.                          |
| F8  | Ведение ставок по депозитам в системе (вместо XLS) с доступом бэк-офиса депозитов и бэк-офиса кредитов; процесс согласования ставки.    | **АЗТ:** кандидат на реализацию в АБС. Учесть разграничение доступа (ИБ: депозиты ≠ кредиты). Заменяет ручной XLS-процесс из Задания 1.                                  |
| F9  | MVP: заявку на открытие депозита обрабатывает менеджер бэк-офиса, подтверждает условия депозита в АБС.                                  | Ручной шаг на MVP — допускает асинхронную (отложенную) обработку заявки (см. R4).                                                                                       |
| F10 | Клиент получает СМС-уведомления после подтверждения размера ставки и после открытия депозита.                                          | Два события уведомления; продьюсер событий — процесс согласования/АБС, потребитель — сервис СМС (F7).                                                                    |
| F11 | **(доп.)** Аутентификация и авторизация клиента в интернет-банке для доступа к персонализированным ставкам и счетам.                    | Персонализация (F5) и работа со счётом (F6) невозможны без идентифицированной сессии. Влияет на безопасность и интеграцию с профилем клиента.                            |
| **U** | **Удобство использования (Usability)**                                                                                              |                                                                                                                                                                         |
| U1  | Максимально удобный интерфейс подачи заявки для клиента на сайте и в интернет-банке.                                                   | Канал привлечения; влияет на конверсию заявок.                                                                                                                          |
| U2  | Соблюдение корпоративной системы дизайна: привычные цвета, брендированные элементы.                                                    | **АЗТ:** ограничение UI-слоя — переиспользование общей дизайн-системы/библиотеки компонентов на сайте и в ИБ.                                                            |
| U3  | **(доп.)** Адаптивность форм (десктоп/мобайл) на сайте и в ИБ.                                                                         | Новые клиенты приходят с сайта с разных устройств; адаптивность напрямую влияет на конверсию MVP.                                                                        |
| **R** | **Надёжность (Reliability)**                                                                                                        |                                                                                                                                                                         |
| R1  | Все сервисы работают 24/7 с доступностью 99,9%.                                                                                        | **АЗТ:** ключевое НФТ (≈8,7 ч простоя в год). Определяет резервирование, отсутствие single point of failure, мониторинг.                                                |
| R2  | Интернет-банк остаётся доступным при сбое в ЦОД и выдерживает нагрузку (сейчас: один ЦОД, одна группа серверов — невыполнимо).         | **АЗТ:** требуется failover на резервный ЦОД и горизонтальное резервирование (см. +R7). Текущее состояние не удовлетворяет R1.                                           |
| R3  | Исключить прямую работу интернет-банка с API АБС в новом процессе.                                                                     | **АЗТ + ограничение (+R5):** БД АБС перегружена, онлайн-поток заявок угрожает работоспособности банка. Развязка ИБ и АБС через промежуточный слой/очередь.                |
| R4  | **(доп.)** Заявки не теряются при недоступности АБС (буферизация и гарантированная доставка до бэк-офиса).                              | Ручная обработка менеджером (F9) допускает асинхронность; буфер (Kafka, см. S3) даёт устойчивость и сглаживает пики, защищая АБС.                                        |
| R5  | **(доп.)** Идемпотентность подачи заявки и подтверждения СМС-кодом (защита от дублей при повторах/ретраях).                            | Финансовая операция: повторное открытие депозита недопустимо. Влияет на дизайн API и ключей идемпотентности.                                                            |
| **P** | **Производительность (Performance)**                                                                                                |                                                                                                                                                                         |
| P1  | Отклик по всем клиентским операциям — миллисекунды.                                                                                    | **АЗТ:** целевая задержка диктует кэширование, денормализацию витрин и отказ от синхронных вызовов АБС в онлайне.                                                        |
| P2  | Устранить медленную загрузку справочных данных (сейчас при платежах справочники грузятся дольше секунды).                              | **АЗТ:** кэш справочников / локальные реплики на стороне ИБ вместо обращения в АБС в реальном времени.                                                                   |
| P3  | Система выдерживает нагрузку онлайн-подачи заявок на большое число продуктов без деградации АБС.                                       | **АЗТ:** throttling и развязка нагрузки (R3, R4); АБС не должна быть на критическом пути онлайн-операций.                                                                |
| **S** | **Поддерживаемость (Supportability)**                                                                                               |                                                                                                                                                                         |
| S1  | Максимально переиспользовать существующие технологии банка: БД MS SQL и Oracle.                                                        | **АЗТ:** ограничение выбора хранилищ — приоритет существующим СУБД.                                                                                                     |
| S2  | Новые технологии допустимы только при совместимости с существующими платформами разработки и наличии внутренней экспертизы.            | **АЗТ:** ограничение стека (вплоть до уровня языков программирования) — снижает риски поддержки.                                                                        |
| S3  | Для очередей сообщений использовать Kafka (на перспективу). Текущая версия платформы ИБ несовместима с Kafka.                          | **АЗТ:** либо апгрейд платформы ИБ / вынос интеграции в новый совместимый сервис, либо адаптер. Прямое включение Kafka в текущий ИБ невозможно.                          |
| S4  | Рассмотреть перевод интернет-банка на микросервисную архитектуру в рамках задачи открытия депозитов.                                   | **АЗТ:** архитектурный сдвиг, ограниченный скоупом MVP; даёт совместимость с Kafka (S3) и горизонтальное масштабирование (+R7).                                          |
| S5  | Разработать документацию для дальнейшего расширения системы.                                                                           | Условие масштабирования продукта и команды; снижает стоимость последующих доработок.                                                                                    |
| S6  | **(доп.)** Наблюдаемость: мониторинг, логирование, алертинг по SLA.                                                                    | Без наблюдаемости невозможно подтверждать и удерживать доступность 99,9% (R1) и контролировать задержки (P1).                                                            |
| **+R** | **+ Ограничения (Restrictions)**                                                                                                  |                                                                                                                                                                         |
| +R1 | Безопасность: шифрование трафика с сайта (передаются чувствительные данные) — TLS/HTTPS.                                               | **АЗТ:** явное требование защиты канала для публичного сайта.                                                                                                           |
| +R2 | Безопасность: шифрование данных при передаче в интернет-банке.                                                                         | **АЗТ:** защита канала ИБ механизмом шифрования.                                                                                                                        |
| +R3 | Безопасность: подтверждение операции открытия депозита СМС-кодом (второй фактор).                                                      | Связано с F6; защита от несанкционированных операций.                                                                                                                   |
| +R4 | Конструктивное: СМС-функционал реализуется силами банка, без доработки ядра подрядчиком.                                               | Ограничение реализации (см. F7) — минимизирует зависимость от подрядчика и сроки.                                                                                       |
| +R5 | Конструктивное: интернет-банк не обращается напрямую к API АБС; интеграция через промежуточный слой/очередь.                           | Ограничение интеграции (см. R3) — защита перегруженной БД АБС и достижение 99,9%.                                                                                       |
| +R6 | Реализационное: стек ограничен совместимыми технологиями + MS SQL/Oracle + Kafka (перспектива) при наличии экспертизы.                 | Сводное ограничение по S1–S3.                                                                                                                                           |
| +R7 | Физическое: два ЦОД (основной + резервный); горизонтальное масштабирование и распределение запросов между серверами, приложениями, ЦОД.| **АЗТ:** целевая инфраструктура отказоустойчивости и масштабирования интернет-банка (см. R2).                                                                           |
| +R8 | Физическое/ограничение: АБС масштабируется только вертикально из-за своей БД.                                                          | **АЗТ:** нельзя закладывать горизонтальное масштабирование АБС — усиливает требование развязки (R3, +R5).                                                                |
| +R9 | **(доп.)** Правовое: офлайн-идентификация новых клиентов (KYC) и защита персональных данных (152-ФЗ) для данных с сайта.               | Банковский регуляторный контекст; формализует ограничение F4 и требования к хранению/передаче ПДн.                                                                      |
| +R10| Ограничение скоупа: MVP охватывает только процесс открытия депозитов; заявку обрабатывает менеджер бэк-офиса вручную.                  | Рамки задачи — определяет минимальный объём интеграций и допускает ручной/асинхронный шаг (F9, R4).                                                                     |
